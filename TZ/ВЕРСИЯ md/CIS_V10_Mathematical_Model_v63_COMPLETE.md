|                                                                                                                                                                                         |

| **Правило 3: Conformal coverage ≠ trading edge**                                                                                                                                        |
|                                                                                                                                                                                         |
| Гарантия покрытия conformal intervals (90%) не означает наличие торгового преимущества. Coverage --- это свойство калибровки неопределённости, не доказательство предсказуемости рынка. |

**4. Multiple Lyapunov Functions --- аналитически верифицировано**

**⊕ Grok→v6.0: Исправлена критическая ошибка Grok: common P не работает для R1-R4. Используем Multiple Lyapunov с ADT.**

**Подход: Multiple Lyapunov Functions с Average Dwell-Time (ADT) условием для switched system.**

**Условие ADT для устойчивости switched system**

V_r(x) = x\^T P_r x, P_r ≻ 0 (regime-specific Lyapunov matrix)

ΔV_r = V_r(x\_{t+1}) - V_r(x_t) ≤ -γ_r \|\|x\|\|² (per-regime decay)

ADT condition: τ_a ≥ ln(μ_switch) / γ_conservative

μ_switch = λ_max_all / λ_min_all = 96.74 / 2.78 = 34.80

τ_ADT ≥ ln(34.80) / 0.05 = 71 шагов \[АНАЛИТИЧЕСКИ ДОКАЗАНО\]

+---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------+
| **Критическое исправление: τ_min ≥ 71, а не ≥ 5 (как у Grok)**                                                                                                                                                                        |
|                                                                                                                                                                                                                                       |
| При быстром переключении режимов (R1→R3→R1 за \<71 шагов) ADT условие нарушается. Система должна применять minimum dwell-time enforcement: запрет переключения режима чаще чем раз в 71 forecast cycle (\~18 часов при 1h прогнозах). |
|                                                                                                                                                                                                                                       |
| Исключение: аварийные переходы (HARD brake при q\>0.55) разрешены немедленно --- это safety override, а не stability switch.                                                                                                          |

>
> \"\"\"Split conformal QR, asymmetric intervals, conservative fallback.\"\"\"
>
> def \_\_init\_\_(self, target_coverage=0.90, max_uncertainty_prior=0.40):
>
> self.target_coverage = target_coverage
>
> self.max_uncertainty_prior = max_uncertainty_prior \# conservative
>
> self.calibration_residuals = \[\]
>
> self.quantile = max_uncertainty_prior \# start conservative (not 0)
>
> self.snapshot_id = None
>
> def calibrate(self, residuals: np.ndarray, snapshot_id: str) -\> float:
>
> n = len(residuals)
>
> if n \>= 30:
>
> \# Finite-sample corrected (Venn-Abers): level = target\*(1+1/n)
>
> level = min(1.0, self.target_coverage \* (1 + 1/n))
>
> self.quantile = np.quantile(residuals, level)
>
> else:
>
> \# CONSERVATIVE fallback --- NOT magic numbers
>
> \# Linear interpolation: prior at n=0, data-driven at n=30
>
> weight = n / 30
>
> dq = np.quantile(residuals, self.target_coverage) if n \> 1 else self.max_uncertainty_prior
>
> self.quantile = (1 - weight)\*self.max_uncertainty_prior + weight\*dq
>
> self.snapshot_id = snapshot_id
>
> return self.quantile
>
> def predict_interval(self, point_estimate: float, residuals_recent=None):
>
> \"\"\"Asymmetric intervals for directional forecast.\"\"\"
>
> alpha = (1 - self.target_coverage) / 2
>
> if residuals_recent is not None and len(residuals_recent) \>= 30:
>
> q_lower = np.quantile(residuals_recent, alpha) \# negative
>
> q_upper = np.quantile(residuals_recent, 1 - alpha) \# positive
>
> else:
>
> q_lower = -self.quantile \# symmetric fallback
>
> q_upper = self.quantile
>
> lower = point_estimate + q_lower
>
> upper = point_estimate + q_upper
>
> coverage_empirical = self.target_coverage \# guaranteed by construction
>
> return lower, upper, coverage_empirical, self.snapshot_id

**Proper Scoring Rules --- три метрики**

Добавлены в feedback loop (Slow Feedback --- Heuristic Contract review):

> def crps_score(y_true, mean, std):
>
> \"\"\"CRPS --- proper scoring for probabilistic forecasts.\"\"\"
>
> z = (y_true - mean) / std
>
> phi = scipy.stats.norm.pdf(z)
>
> Phi = scipy.stats.norm.cdf(z)
>
> return std \* (z\*(2\*Phi-1) + 2\*phi - 1/np.sqrt(np.pi))
>
> def pinball_loss(y_true, q_hat, alpha):
>
> \"\"\"Pinball --- proper for quantile forecasts.\"\"\"
>
> e = y_true - q_hat
>
> return e\*alpha if e \>= 0 else e\*(alpha-1)
>
> def winkler_score(y_true, lower, upper, alpha=0.10):
>
> \"\"\"Winkler --- proper for prediction intervals.\"\"\"
>
> width = upper - lower
>
> penalty = (2/alpha)\*max(0, lower-y_true) + (2/alpha)\*max(0, y_true-upper)
>
> return width + penalty

**34. Uncertainty Escalation Rule**

**u_width и coverage_drift теперь явно влияют на decision speed и escalation policy.**

**42. Silence Debt Metric**

**Диагностическая метрика второго порядка. Smoke detector, не actuator. Не влияет на D_t, c_min, q_t, w_i.**

S_debt(t+1) = rho_H \* S_debt(t) + 1\[Y_t = SILENCE_INVALID\]

rho_H = 1 - 1/H_verify (привязка к горизонту верификации)

**43. Feasibility Severity Score**

**Weighted severity score как усиление count-rule. Count-rule сохраняется как safety baseline.**

B_sev = sum_i w_i \* 1\[C_i violated\]

  ---------------------------------------------------------- ----------------------- ----------------------------------------------------------------
  **Условие**                                                **Вес w_i**             **Обоснование**

  C1: Stability breach (V_r(x) \> V_max)                     3.0                     Математическая основа системы нарушена --- наивысший приоритет

  C2: Observability gap (N_priors\<2 или eps_error\>=0.10)   1.0                     Мониторинг ухудшен, система работает --- меньший вес

  C3: Latency breach (l\>200ms или l_forecast\>30s)          2.0                     Операциональное ограничение --- деградированный режим

  C4: Conformal invalid (\|coverage-0.90\|\>=0.05)           2.5                     Целостность uncertainty intervals --- доверие к прогнозу

  C5: ADT violation (tau \< 71 при плановом switch)          3.0                     Гарантия устойчивости --- тот же уровень что C1

  C6: Controller conflict (conflict_score \> 0)              2.0                     Governance --- сопоставимо с операциональными ограничениями
  ---------------------------------------------------------- ----------------------- ----------------------------------------------------------------

Max B_sev = 13.5 (все условия нарушены одновременно).

  ----------------------- ---------------------------------------- -----------------------------------------------------------------------------
  **B_sev диапазон**      **Действие**                             **Отношение к count-rule**

  B_sev \< 2.0            Нормально --- нет действий               0-1 нарушений (обычно)

  2.0 \<= B_sev \< 4.0    Monitoring alert (elevated)              1-2 нарушения средней тяжести

  4.0 \<= B_sev \< 6.0    Early Warning активируется               Эквивалент count\>=3 при средних весах; C1 или C5 в одиночку дают B_sev=3.0

  B_sev \>= 6.0           Немедленный переход R3 + Early Warning   C1+C5 вместе=6.0; или 3+ нарушения тяжёлых
  ----------------------- ---------------------------------------- -----------------------------------------------------------------------------

+---------------------------------------------------------------------------------------+
| **Count-rule сохраняется как safety baseline (Invariant I-14)**                       |
|                                                                                       |
| count\>=3 violations -\> R3 + Early Warning (нижняя граница, всегда активна).         |
|                                                                                       |
| B_sev \>= 4.0 -\> Early Warning (более тонкий слой: C1 или C5 в одиночку достаточно). |
|                                                                                       |
| **Escalation срабатывает при выполнении ЛЮБОГО из двух правил.**                      |

**47. Transaction Boundary Clarification**

**Устраняет скрытую post-commit mutation: Step 15 вынесен из Tx_Forecast в отдельный Tx_Feedback. Два типа транзакций --- два разных объекта.**

+-------------------------------------------------------------------------------------------------------------+
| **Forecast transaction and feedback transaction are different transactional objects.**                      |
|                                                                                                             |
| Tx_Forecast(t): Steps 1-14. Commit point = Step 14. Содержит прогноз, state snapshot и audit bundle.        |
|                                                                                                             |
| Tx_Feedback(t→t+1): Step 0 следующего цикла. Применяет fast feedback к состоянию ДО Step 2 snapshot freeze. |
|                                                                                                             |
| **Step 14 --- честный atomic commit. Состояние на момент выдачи прогноза полностью зафиксировано.**         |
+-------------------------------------------------------------------------------------------------------------+

**Пересмотренная последовательность цикла**

  -------------- ------------------------------- ----------------------- ------------------------------------------------------------------------------------------------------------------------------------------------------------- --------------
  **Шаг**        **Название**                    **Транзакция**          **Действие**                                                                                                                                                  **Класс**

  Step 0         PRE-CYCLE FAST FEEDBACK APPLY   Tx_Feedback(t-1→t)      Применить verified outcomes из предыдущего verification window: обновить w_i, I_t, mu_t. Завершить ДО Step 2. Если нет дозревших outcomes --- Step 0 no-op.   CRITICAL

  Step 1         DATA INGEST                     Tx_Forecast(t) начало   Gate evaluation, stale cache check, stale_age update                                                                                                          CRITICAL

  Step 2         SNAPSHOT READ                   Tx_Forecast(t)          Atomic read ПОСЛЕ Step 0. Freeze всех 8 determinism params. Снапшот уже включает post-feedback state.                                                         CRITICAL

  Steps 3-13     FORECAST COMPUTE                Tx_Forecast(t)          Regime, Shock, Epistemic, Scoring, Conformal UQ, Uncertainty Escalation, D_t, Y_t, Feasibility, S_debt, E_ctrl                                                CRITICAL+OBS

  Step 14        ATOMIC WRITE (COMMIT)           Tx_Forecast(t) конец    Единый commit: forecast + state + audit + H(Pi_t) + Y_t + filter_cause + S_debt + B_sev + E_ctrl_delta + X_exec + exec_outcome_reason                         CRITICAL

  Step 15        DEFERRED QUEUE APPEND           Не транзакция           Только append slow FB tasks в очередь. Исполнение отдельным процессом.                                                                                        OBSERVABLE
  -------------- ------------------------------- ----------------------- ------------------------------------------------------------------------------------------------------------------------------------------------------------- --------------

+------------------------------------------------------------------------------------------------------------------------------------------------------------------+
| **Почему Решение B --- единственно правильное для sealed модели**                                                                                                |
|                                                                                                                                                                  |
| Детерминизм: H(Pi_t) фиксирует state в Step 2. При Решении A Step 15 менял бы weights ПОСЛЕ коммита --- committed state расходился бы с реальным состоянием.     |
|                                                                                                                                                                  |
| Control Theory: не \"лишняя задержка\" а правильная причинность. Цикл t выдаёт прогноз; цикл t+1 использует обновлённое состояние после подтверждённых outcomes. |
|                                                                                                                                                                  |
| Audit clarity: ABORTED/REJECTED в X_exec однозначно описывают состояние без путаницы pre/post-feedback.                                                          |

**48. Runtime Precedence Order**

**Глобальный порядок применения ограничителей при одновременном срабатывании нескольких. Снимает архитектурную неоднозначность для разработчиков.**

  ----------------- ---------------------------------------------- ---------------------------------------------------- ------------------------------------------------------------------------------------------
  **Приоритет**     **Механизм**                                   **Условие применения**                               **Эффект**

  P1 (highest)      Hard Invariant violation (I-01..I-17)          Любое нарушение I-01..I-17 в любой момент цикла      Abort cycle или clamp. Не продолжать без explicit handling.

  P2                Determinism mismatch H(Pi_t)                   H(Pi_t) mismatch при Step 14                         REJECTED + exec_outcome_reason=determinism_mismatch. Invariant I-13.

  P3 (exception)    External absolute hard override                Hack/regulatory/external event asserted externally   Short-circuits P4-P9. Preempts data path selection. Immediate HARD-SUPPRESSED.

  P4                Data path resolution                           Steps 1-2: data freshness check                      fresh -\> продолжить. stale и age\<=ttl -\> FALLBACK path. stale и age\>ttl -\> ABORTED.

  P5                Hard brake / safety brake                      q_t \> 0.55 OR q_t в HARD zone (hysteresis)          D_t=HARD-SUPPRESSED. Неотменяемо кроме P3.

**52. Step 0 Matured-Outcome Selection Rule**

**Явный контракт для Tx_Feedback(t→t+1). Снимает кодовую неоднозначность Task 24: что считается дозревшим, как агрегировать, что делать если дозрело несколько.**

+------------------------------------------------------------------------------------------------------------------------------------------------+
| **Step 0 --- контракт выбора и применения matured outcomes**                                                                                   |
|                                                                                                                                                |

+------------------------------------------------------------------------------------------------------------------------------------------------+

  ----------------------- --------------------------------------------------------------------------------------------------------------------------------------------------------- -----------------------------------------------------------------
  **Правило**             **Формулировка**                                                                                                                                          **Обоснование**

  Selection unit          Один verification batch = все outcomes с elapsed(forecast_timestamp) \>= H_verify в момент начала Step 0                                                  Атомарность батча --- не разрывать логически связанные outcomes

  Batching rule           Обрабатывать ВСЕ дозревшие outcomes доступные в начале Step 0. Не откладывать частичные батчи на следующий цикл.                                          Полнота обновления --- нет selective application

  Aggregation rule        Если несколько outcomes дают конфликтующие весовые сигналы: Δw_i = mean(Δw_i\^k, k in batch). Результирующий Δw_i подлежит MAX_WEIGHT_DELTA constraint.   Устойчивость при одновременных противоположных сигналах

  Idempotency rule        Каждый outcome_id обрабатывается не более одного раза. Повторная обработка того же outcome_id = no-op (idempotency_key в Tx_Feedback).                    Защита от double-apply при retry

  Abort rule              Если Step 0 не завершён из-за timeout → пропустить весь батч. Outcomes остаются в очереди для следующего цикла. НЕ применять частичный батч.              Атомарность Step 0 как транзакции
  ----------------------- --------------------------------------------------------------------------------------------------------------------------------------------------------- -----------------------------------------------------------------

Step 0 не является частью Tx_Forecast. Это отдельная Tx_Feedback(t-1→t). Её abort не влияет на Tx_Forecast(t). H(Pi_t) фиксируется в Step 2 уже после завершения Step 0.

**54. Hysteresis Admissible Region**

**Formal constraints для CONFIG/threshold_hysteresis_policy.json. Параметры вне admissible region могут нарушить ADT logic, precedence semantics или safety guarantees.**

+--------------------------------------------------------------------------------------------------------------------------------------------------------------------+
| **Принцип: модель задаёт форму и constraints. CONFIG задаёт числа в допустимой области.**                                                                          |
|                                                                                                                                                                    |
| Параметры вне admissible region недопустимы даже если они технически записываемы в JSON. Валидатор CONFIG обязан проверять все 5 constraints при каждом изменении. |
+--------------------------------------------------------------------------------------------------------------------------------------------------------------------+

  ------------------------------------- ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- ---------------------------------------------------------------------------------------------------------- -------------------------------------------------------
  **Constraint**                        **Правило**                                                                                                                                                                                                       **Нарушение**                                                                                              **Обоснование**

  C-Hyst-1 (gap)                        exit_threshold \< enter_threshold. delta_hyst = enter - exit \> 0. Минимум: delta_hyst_min \>= 0.02 для q-thresholds, \>= 0.01 для u-thresholds.                                                                  Если exit \>= enter: система немедленно покидает зону как только входит -\> hysteresis не работает.        Hysteresis requires strict gap between enter and exit

  C-Hyst-2 (ordering)                   Для одной и той же метрики с несколькими уровнями: enter_L(k+1) \> enter_L(k) + delta_hyst_min. Нет пересекающихся зон. Нет инверсии порогов.                                                                     Если зоны пересекаются: система может находиться в двух уровнях одновременно.                              No ambiguous zone membership

  C-Hyst-3 (ADT compatibility)          min_persist \>= 1 cycle (всегда). Для q-thresholds: min_persist \<= tau_ADT / 2 = 35 cycles. Для u-thresholds: min_persist \<= 10 cycles.                                                                         Если min_persist \>= tau_ADT: hysteresis может блокировать safety transitions требуемые ADT enforcement.   Hysteresis must not prevent ADT-required switches

  C-Hyst-4 (precedence compatibility)   Hard brake (P5): min_persist = 1 (немедленно, без исключений). Feasibility safety actions P6a: не могут быть задержаны hysteresis (delay=0). Никакой hysteresis параметр не может заставить P1-P3 checks ждать.   Если safety thresholds имеют min_persist \> 1: возможна задержка safety response.                          Safety thresholds must not be debounced

  C-Hyst-5 (exit completeness)          Каждый enter threshold обязан иметь соответствующий exit threshold. Исключение: HARD brake --- exit только через R5 path (exit threshold запрещён).                                                               Если exit threshold отсутствует: система навсегда остаётся в зоне после входа.                             Every entered zone must have an exit path
  ------------------------------------- ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- ---------------------------------------------------------------------------------------------------------- -------------------------------------------------------

**Validation rule (для CONFIG validator)**

FOR each threshold_group IN hysteresis_policy:

ASSERT exit \< enter (C-Hyst-1)

ASSERT delta_hyst \>= delta_hyst_min (C-Hyst-1)

ASSERT no zone overlap (C-Hyst-2)

ASSERT 1 \<= min_persist \<= tau_ADT/2 (C-Hyst-3)

IF safety_threshold: ASSERT min_persist == 1 (C-Hyst-4)

ASSERT exit_defined OR is_hard_brake (C-Hyst-5)

Нарушение любого constraint → CONFIG rejected. Heuristic config изменение с нарушенным constraint не применяется даже при наличии change_id.

> --- Step 14 --- единственная точка записи forecast bundle. До Step 14 ничего не записывается постоянно.
>
> --- Step 15 НЕ входит в Tx_Forecast. Он только добавляет задачи в очередь.
>
> --- Step 0 завершается ДО Step 2. Snapshot read захватывает post-feedback state.
>
> --- При abort Tx_Forecast --- Step 0 outcomes остаются в очереди для следующего цикла.

**Canonical temporal sequence (Y_t / P6a / X_exec)**

> --- Y_t вычисляется в Step 10 --- до P6a.
>
> --- При force_neutral: D_t_final=NEUTRAL, Y_t пересчитывается с новым D_t_final.
>
> --- При reject_output или hard_override: Y_t=DIR_INVALID (override, без пересчёта).
>
> --- X_exec фиксируется при Step 14 commit --- после P6a применения.
>
> --- НИКОГДА: P6a не применяется до Step 9 (до D_t).

**Steps 5-8 degraded commit**

> --- Step 6 timeout = ABORTED. Никогда не COMMITTED degraded.
>
> --- Step 7 timeout: u_low = c_t - 0.40, u_high = c_t + 0.40. conformal_degraded=True.
>
> --- Step 8 timeout: unc_level = L2. unc_escalation_degraded=True.
>
> --- Step 5 timeout: gamma_t = gamma_t_prev, eps_t = eps_t_prev. epistemic_degraded=True.

**Hysteresis CONFIG validation**

> --- exit_threshold \< enter_threshold для каждого threshold (C-Hyst-1).
>
> --- delta_hyst \>= 0.02 для q-thresholds, \>= 0.01 для u-thresholds.
>
> --- min_persist в диапазоне \[1, 35\] для q-thresholds (C-Hyst-3).
>
> --- Hard brake: min_persist = 1 (немедленно, нет исключений).
>
> --- Каждый enter имеет exit кроме HARD brake (C-Hyst-5).
