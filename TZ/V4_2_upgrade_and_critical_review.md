# Часть 1 — Upgrade: V4.1 → V4.2

## Цель V4.2
Усилить V4.1 не за счёт расширения функционала, а за счёт:
- устойчивости
- воспроизводимости
- защиты от ошибок
- инженерной надёжности

---

## 1. Добавленные обязательные модули

### 1.1 Data Quality Layer

**Файл:** ENGINE/data_quality_monitor.py

**Задача:**
Контроль качества данных до их использования.

**Что добавляет:**
- проверка пропусков (missing candles)
- контроль NaN
- детекция выбросов
- контроль разрывов во времени

**Почему критично:**
Любая ошибка в данных = ложные паттерны

**Статус:**
- НОВОЕ в V4.2
- Закрывает пробел V4.1

---

### 1.2 DSL Test Suite

**Файл:** ENGINE/dsl_test_suite.py

**Задача:**
Проверка корректности Pattern DSL

**Что добавляет:**
- unit-тесты паттернов
- эталонные кейсы
- sandbox исполнение

**Почему критично:**
DSL = источник скрытых ошибок

**Статус:**
- НОВОЕ в V4.2

---

### 1.3 Baseline Engine

**Файл:** ENGINE/baseline_engine.py

**Задача:**
Сравнение паттернов с простыми стратегиями

**Что добавляет:**
- buy & hold baseline
- random baseline
- naive strategies

**Почему критично:**
Без baseline нет понимания “есть ли edge”

**Статус:**
- НОВОЕ в V4.2

---

### 1.4 Feature Versioning

**Папка:** DATA/features_snapshots/

**Задача:**
Версионирование данных

**Что добавляет:**
- snapshot_id
- data_hash
- воспроизводимость экспериментов

**Почему критично:**
Без этого невозможно повторить результат

**Статус:**
- РАСШИРЕНИЕ Feature Store

---

### 1.5 Pattern Risk Scoring

**Файл:** ENGINE/pattern_risk.py

**Задача:**
Оценка риска паттерна

**Метрика:**
risk_score = volatility + regime_dependency + decay_rate

**Почему критично:**
confidence ≠ риск

**Статус:**
- НОВОЕ

# Часть 2 — Критический обзор реализации

---

# 2.1 Главная проблема: это не проект, а система систем

Это не:
- скрипт
- бот
- аналитический инструмент

Это:
> **исследовательская платформа уровня mini-hedge-fund R&D**

---

# 2.2 Основные риски реализации

## Риск 1 — Перегрузка архитектуры

Система включает:
- data layer
- research engine
- knowledge system
- ML-like model
- UI
- ingestion pipelines

👉 Риск:
разработка остановится из-за сложности

---

## Риск 2 — Ложное чувство прогресса

Claude + UI могут создавать иллюзию работы системы

👉 Но:
- без validation engine всё фейк
- без data quality всё фейк

---

## Риск 3 — DSL как точка отказа

DSL:
- сложный
- чувствителен к ошибкам
- влияет на всё

👉 Один баг = сломанные выводы

---

## Риск 4 — Data Dependency

Система критично зависит от:
- Binance API
- внешних данных
- корректности расчётов

👉 Любая проблема = деградация системы

---

## Риск 5 — Переоценка ML-части

Live Model:
- выглядит как ML
- но это rule-based система

👉 Риск:
ожидание “магии”, которой нет

---

# 2.3 Самое сложное место проекта

НЕ:
- UI
- ingestion
- Claude

А:

> **Validation + Data + DSL вместе**

Это ядро системы

---

# 2.4 Где реально сломается проект (если сломается)

1. На этапе DSL
2. На этапе данных
3. На этапе validation discipline
4. При попытке сделать всё сразу

---

# 2.5 Как НЕ провалить проект

## Правило 1
Сначала:
- схемы
- validation
- данные

Потом:
- всё остальное

---

## Правило 2
Запускать только Этап 1–2

НЕ трогать:
- модель
- новости
- trust

---

## Правило 3
Каждый паттерн = проверка

Без проверки:
- не сохранять
- не использовать

---

## Правило 4
Не верить Claude

Claude:
- помощник
- не источник истины

---

## Правило 5
Делать маленькие победы

НЕ:
“вся система сразу”

А:
- гипотеза → тест → результат

---

# 2.6 Реальный сценарий успеха

Успех проекта =

1. Есть 10–20 validated patterns  
2. Есть негативные знания  
3. Есть reproducible pipeline  
4. Есть честная статистика  

Только после этого:
→ модель
