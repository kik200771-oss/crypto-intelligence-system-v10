## L-01: TASK может расходиться с ТЗ — проверять первоисточник

**Когда возникло:** TASK_03 v1 (2026-04-17)
**Категория:** architecture

**Симптом (что мы увидели):**
TASK_03 v1 предписывал реализовать `orchestrate(request)` с 6 входами и per-input timeout matrix (BLOCK/DEGRADE/SKIP). Это было исполнено, закоммичено, помечено `context_orchestrator: ready`. Но при ревью архитектор обнаружил что в ТЗ V10.0-r1 Задача 3 требует совсем другого: публичный API `build_context(query, symbol, timeframe) → str`, Task Awareness на 4 профиля, бюджет 8000 токенов, Fast Lane Invariant. Реализованный код был Input Assembler (Task 21 по ТЗ), а не Context Orchestrator (Task 3).

**Причина (почему так произошло):**
TASK файл был сгенерирован другим агентом без сверки с каноном ТЗ V10.0-r1. Claude Code добросовестно его выполнил, но TASK сам по себе был неправильный.

**Правильный подход:**
Перед началом задачи — если чувствуешь что-то "не то" (интерфейс странный, требования не вписываются в общую картину) — **открой ТЗ и сверь сам**. Если расхождение есть — § 3 (СТОП), не продолжай выполнять неправильный TASK.

**Правило на будущее:**
При любом сомнении в соответствии TASK ↔ ТЗ — открой `TZ/CIS_TZ_V10_0_r1.md`, найди соответствующий раздел Задачи, сверь. При расхождении — СТОП.

**Применимость:** все задачи. Особенно задачи MDC (first live).

## L-02: Status "ready" должен отражать реальное соответствие ТЗ, а не факт написания кода

**Когда возникло:** аудит (2026-04-18)
**Категория:** workflow

**Симптом (что мы увидели):**
`component_status.json` содержал 4 компонента со статусом `ready`. При ревью оказалось: 0 из 4 реально готовы по ТЗ. `initialize_system` создал 5 слоёв из 8, `schema_layer` — 3 схемы из 6, `data_quality_gates` — Gate 6 не тот, `context_orchestrator` — сделал Input Assembler вместо Context Orchestrator.

**Причина (почему так произошло):**
Статус "ready" ставился по факту "код написан и тесты пройдены". Но "тесты пройдены" не значит "код соответствует ТЗ" — тесты могли проверять неправильное поведение.

**Правильный подход:**
Статус `ready` = **код соответствует ТЗ + тесты из TASK пройдены + все критерии готовности из TASK выполнены**. Тесты проверяют требования TASK, а TASK сверен с ТЗ (L-01).

Доступные статусы и их значения:
- `not_started` — ещё не начат
- `in_progress` — пишется код
- `needs_rewrite` — код написан, но не соответствует ТЗ
- `needs_rebuild` — код частично готов, требует значительной доработки
- `ready` — полностью соответствует ТЗ, все тесты PASS

**Правило на будущее:**
Не ставь `ready` автоматически. Перед изменением статуса пройди чеклист § 24 и сверь критерии готовности с TASK.

**Применимость:** все задачи.

## L-03: Никаких hardcoded абсолютных путей в коде модулей

**Когда возникло:** аудит (2026-04-18)
**Категория:** code-quality

**Симптом (что мы увидели):**
В `MARKET_MIND/ENGINE/context_orchestrator.py:22` был hardcoded `r'C:\CODE\MARKET ANALYSIS\MARKET_MIND\LAYER_H_INTERFACE\logs'`. При переезде проекта с `C:\КОДИНГ\...` на `C:\CODE\...` часть файлов обновилась, часть — нет. Некоторые скрипты ломаются в зависимости от того где запустить.

**Причина (почему так произошло):**
Удобство быстрого прототипирования: "пропишу путь, запустится у меня". Забыли что проект переедет.

**Правильный подход:**
Используй относительные пути через `Path(__file__).resolve()`:

```python
from pathlib import Path

class ContextOrchestrator:
    def __init__(self, base_path: str | None = None):
        if base_path is None:
            # MARKET_MIND/ENGINE/context_orchestrator.py → MARKET_MIND/
            base_path = Path(__file__).resolve().parent.parent
        self.base = Path(base_path)
        self.logs_dir = self.base / "LAYER_H_INFRA" / "logs"
```

**Правило на будущее:**
В коде `MARKET_MIND/ENGINE/*.py` — никаких `C:\`, `D:\`, `/home/`, `/Users/`. Только `Path(__file__).resolve().parent.parent` или аналог. Допустимый аргумент `base_path` в конструкторе для тестов.

**Применимость:** все модули в `MARKET_MIND/ENGINE/` и `scripts/`.

## L-04: Emoji в print() ломает вывод на Windows cp1251

**Когда возникло:** тесты TASK_03 v1 (2026-04-17)
**Категория:** code-quality

**Симптом (что мы увидели):**
Тесты Python выводили `print(f'\u2705 Test 1 OK: ...')`. На Windows с дефолтной кодировкой cp1251 скрипт падал с `UnicodeEncodeError: 'charmap' codec can't encode character '\u2705'`.

**Причина (почему так произошло):**
Windows Python по умолчанию использует кодировку активной кодовой страницы (cp1251 для русской локали), которая не поддерживает emoji.

**Правильный подход:**
В тестовых скриптах использовать ASCII-маркеры:
- `[OK]` вместо `✅`
- `[FAIL]` вместо `❌`
- `[WARN]` вместо `⚠️`
- `[PASS]` / `[FAIL]` для итоговых статусов

Если категорически нужен Unicode в print — в начале скрипта:
```python
import sys
if sys.stdout.encoding and sys.stdout.encoding.lower() != "utf-8":
    sys.stdout.reconfigure(encoding="utf-8")
```

**Правило на будущее:**
В **любом** `.py` файле который будет запускаться (тесты, CLI, main-блоки) — **никаких emoji в print**. Только ASCII-маркеры. В документации (`.md`, docstring) emoji допустимы.

**Применимость:** все `.py` файлы с `print()`.

## L-05: GitHub токены не хранятся в `.env`, только в Windows Credential Manager

**Когда возникло:** аудит (2026-04-18)
**Категория:** security

**Симптом (что мы увидели):**
В `.env` проекта хранилась строка `GITHUB_TOKEN=ghp_REDACTED_CLASSIC_TOKEN_WAS_HERE_DO_NOT_RESTORE` — реальный рабочий classic-токен. При отправке zip-архива на ревью токен попал в переписку с архитектором. `.gitignore` спас от утечки в публичный git, но сама практика опасная.

**Причина (почему так произошло):**
Токен был создан для автоматизации GitHub и сохранён в `.env` "для удобства". Забыли что `.env` может попасть в zip, в логи, в bug report, в скриншоты.

**Правильный подход:**
Для `git push` используется Windows Credential Manager (Git сам туда лезет при первом push). Токен в `.env` **не нужен** для push.

Если токен нужен для API-вызовов (создание репо, webhooks и т.п.) — **временно** через переменную окружения при запуске скрипта, не в `.env`.

Для fine-grained токенов использовать минимальные permissions:
- Repository access: только нужный репо
- Permissions: `Contents: Read and write`, `Metadata: Read` — и всё
- Expiration: 90 days максимум

**Правило на будущее:**
- Никогда не помещай `GITHUB_TOKEN` в `.env`
- Никогда не принтуй содержимое `.env` целиком в логи / отчёты
- При подозрении на утечку — немедленно отозвать токен на GitHub
- В zip-архивах для ревью — явно исключать `.env` (проверить в `scripts/create_audit_zip.py`)

**Применимость:** работа с GitHub, любые секреты.

## L-07: Stub-данные в production-коде скрывают реальное состояние модуля

**Когда возникло:** TASK_03 v1 (2026-04-17)
**Категория:** architecture

**Симптом (что мы увидели):**
В `context_orchestrator.py` методы `_get_feature_snapshot`, `_get_validated_patterns`, `_get_regime_context` возвращали hardcoded "тестовые" данные (rsi=45.2, confidence=0.72 и т.д.) с полем `"stub": True`. Это создавало иллюзию что система работает. При интеграции в Model Core это дало бы искусственно стабильный прогноз, не отражающий реальность.

**Причина (почему так произошло):**
Удобство: "модули ещё не написаны, дам заглушки чтобы оркестратор запускался". Но заглушки оказались неотличимы от реальных данных для вышестоящих модулей.

**Правильный подход (graceful degradation):**
Если источник данных недоступен — метод возвращает `None`, а вышестоящий код явно проверяет:
- Если `None` и этот блок обязательный → `context_degraded = True`, блок помечается missing
- Если `None` и блок опциональный → просто не включается в контекст

Тестовые данные создавать **только** в test fixtures, в отдельных файлах `tests/fixtures/*.json`, и только для тестов — не в production-коде.

**Правило на будущее:**
В `MARKET_MIND/ENGINE/` методы чтения внешних данных либо возвращают реальные данные, либо `None`. Никогда не выдумывают значения. Если источник ещё не реализован — метод возвращает `None` и пишет warning в лог.

**Применимость:** все Data Access методы, Feature Store, Context Orchestrator, всё что читает данные из других слоёв.

## L-08: Fast Lane Invariant — прогноз 1h/4h никогда не блокируется

**Когда возникло:** TASK_03 v1 (2026-04-17)
**Категория:** architecture

**Симптом (что мы увидели):**
В `context_orchestrator.py` при timeout BLOCK-входа возвращался `status: "ABORTED"`. Это значит Fast Lane прогноз (1h/4h) мог не вернуться пользователю, если один из источников данных "тормозит".

**Причина (почему так произошло):**
Буквальное чтение слова "BLOCK" в per-input timeout matrix как "блокировать прогноз". Но ТЗ V10.0-r1 явно прописывает **Fast Lane Implementation Invariant**:

> Fast Lane (1h/4h) numeric forecast path must NEVER wait for Claude response. Context Orchestrator обязан: (1) отдавать Fast Lane прогноз детерминированно без вызова Claude API, (2) запускать Claude slow path асинхронно post-forecast, (3) никогда не блокировать 1h/4h прогноз ожиданием KB / explainability / prior review. **Нарушение этого инварианта является архитектурной ошибкой.**

И далее по Failover Rule:
> При timeout — прогноз строится без prior_weight_factor (prior_snapshot_id=null), пометка context_degraded=True в prediction_schema. **Прогноз НЕ блокируется — это нарушало бы Fast Lane Invariant.**

**Правильный подход:**
В Fast Lane при любом timeout:
- `context_degraded = True`
- Missing данные помечаются, но не блокируют
- Возвращается минимальный валидный контекст (header + query + что успели собрать)
- Статус `"OK"` или `"DEGRADED"`, но **никогда не `"ABORTED"`** в Fast Lane

ABORT допустим **только** в Slow Lane (24h / research), где latency не критична.

**Правило на будущее:**
Перед любой реализацией timeout/failover логики в модулях Fast Lane pipeline (Context Orchestrator, Input Assembly, Model Core, Prior Manager) — проверь что код **не блокирует** возврат прогноза. Если ловишь себя на написании `return status="ABORTED"` в Fast Lane — это сигнал вернуться к ТЗ раздел 1.20 (Fast Lane / Slow Lane) и разделу "Fast Lane Implementation Invariant" из Задачи 3.

**Применимость:** Задачи 3, 6, 14, 21, 24, 29 — всё что на критическом пути Fast Lane.

## L-09: Предусловия TASK читаются буквально — формулировать с учётом preload-файлов

**Когда возникло:** TASK_01 (2026-04-18)
**Категория:** workflow

**Симптом (что мы увидели):**
TASK_01 имел предусловие "1. `git status` → рабочая директория чистая, нет uncommitted changes". Сергей перед запуском задачи вручную положил 4 файла когнитивной системы (CLAUDE.md, LESSONS_LEARNED, PATTERNS, ANTIPATTERNS) в рабочую директорию — это было частью задачи. Claude Code прочитал предусловие буквально, увидел `git status` с modified/untracked файлами и корректно остановился по § 3.

Это **не ошибка Claude Code** — это корректное применение § 3. Ошибка была в формулировке TASK: предусловие не учитывало что пользователь может preload-нуть файлы для задачи.

**Причина (почему так произошло):**
Архитектор формулировал предусловия на основе общего workflow "начинаем с чистого состояния", но для TASK_01 workflow включал ручной preload файлов от Сергея. Формулировка не различала "посторонние изменения" от "нужных для задачи файлов".

**Правильный подход:**
В TASK, где пользователь вручную кладёт файлы перед выполнением (pre-load scenario), формулировать предусловия точно:

Плохо:
> Предусловие 1: `git status` → рабочая директория чистая, нет uncommitted changes

Хорошо:
> Предусловие 1: `git status` показывает **только** следующие изменения:
> - modified: CLAUDE.md (если заменили на новую версию)
> - untracked: LESSONS_LEARNED.md, PATTERNS.md, ANTIPATTERNS.md (положены Сергеем)
> - untracked: TASKS/ACTIVE/TASK_NN_*.md (сам файл задачи)
>
> Других изменений быть не должно. Все эти файлы будут закоммичены в Части N.

**Правило на будущее:**
Если архитектор — при написании TASK'а уточняй предусловия с учётом preload-файлов. Если исполнитель — при обнаружении "лишних" изменений сверяй их со списком preload-файлов в TASK. Если файлы совпадают с preload — это не нарушение предусловия. Если не совпадают — § 3.

**Применимость:** все TASK где архитектор передаёт файлы через pre-load (не через текст TASK).

## L-10: Реальные секреты в обучающих примерах ломают GitHub push protection

**Когда возникло:** TASK_01 (2026-04-18)
**Категория:** security

**Симптом (что мы увидели):**
В `ANTIPATTERNS.md` (AP-05) и `LESSONS_LEARNED.md` (L-05) архитектор использовал реальный (хоть и отозванный) classic GitHub токен как обучающий пример: показать как "неправильный" код выглядел. При попытке push в main, GitHub push protection обнаружил паттерн валидного токена и заблокировал push с ошибкой.

Токен был отозван ДО использования в документации, поэтому утечки не произошло. Но push не проходил пока токен не будет удалён или заменён.

**Причина (почему так произошло):**
Желание сделать пример "живым" и запоминающимся через реальный артефакт. GitHub push protection не различает "живой токен" и "отозванный токен" — реагирует на формат.

**Правильный подход:**
В обучающих примерах, документации, тестовых данных — **только явные маркеры** которые синтаксически не могут быть валидным токеном:

```
GITHUB_TOKEN=ghp_REDACTED_CLASSIC_TOKEN_WAS_HERE_DO_NOT_RESTORE
BINANCE_API_KEY=FAKE_KEY_FOR_DOCUMENTATION
ANTHROPIC_API_KEY=sk-ant-REDACTED-EXAMPLE
```

Ключевые слова `REDACTED`, `EXAMPLE`, `FAKE`, `DO_NOT_RESTORE` — сигнал читателю что это не настоящее значение, и сигнал детекторам секретов что это не утечка.

**Правило на будущее:**
В документации / комментариях / тестах НИКОГДА не использовать реальные значения секретов — даже отозванные, даже прошедшие rotation. Всегда явные маркеры вместо.

**Применимость:** все `.md`, docstrings, test fixtures, примеры в TASK файлах.

## L-11: `git commit --amend` как чистый способ убрать секрет из последнего коммита

**Когда возникло:** TASK_01 (2026-04-18)
**Категория:** workflow

**Симптом (что мы увидели):**
При push ветки `task-01-cognitive-foundation` GitHub push protection заблокировал из-за реального токена в последнем коммите (см. L-10). Нужно было:
- Не загрязнять историю лишним коммитом "fix: remove token"
- Не делать force push в main (запрещено — ломает историю для других)
- Сохранить весь остальной diff последнего коммита нетронутым

**Причина выбора решения:**
`git commit --amend` переписывает **последний** коммит в текущей ветке. Если ветка ещё не опубликована или опубликована только в своей ветке (не в main) — это безопасно.

**Правильный подход — последовательность:**

```bash
# 1. Исправить файлы (заменить реальный токен на REDACTED-маркер)
# 2. Добавить изменения
git add ANTIPATTERNS.md LESSONS_LEARNED.md

# 3. Переписать последний коммит с этими изменениями (без создания нового)
git commit --amend --no-edit

# 4. Проверить что в истории ветки токена больше нет
git log --all -S "ghp_LEAKED_PATTERN" --oneline
# Должно быть пусто

# 5. Force push обязателен (хеш коммита изменился), но --force-with-lease безопаснее:
git push origin <branch-name> --force-with-lease

# 6. Только ПОСЛЕ успешного push в ветку — merge в main обычным путём
git checkout main
git merge <branch-name>  # fast-forward
git push origin main      # обычный push, без force
```

**Правило на будущее:**
- `git commit --amend` применяй только к **последнему** коммиту, только в **своей ветке**, только **до merge в main**.
- После amend — `--force-with-lease` (не `--force`), чтобы не перезаписать случайные чужие изменения.
- В main force push **запрещён** всегда. Если секрет ушёл в main — отдельная процедура через git filter-repo / BFG Repo Cleaner и координация с архитектором (§ 3).

**Применимость:** чистка последнего коммита от секретов/опечаток/забытых файлов в личной ветке.

## L-13: GitHub UI web-страница может быть закэширована — не единственный источник истины о remote

**Когда возникло:** мини-аудит 2026-04-18 (после TASK_01)
**Категория:** workflow

**Симптом (что мы увидели):**
Архитектор после отчётов TASK_00 и TASK_01 провёл мини-аудит через `web_fetch` на главную страницу репо `https://github.com/<user>/<repo>`. Страница показала:
- Repo имеет "3 Commits" (хотя в remote main было 8 коммитов)
- В корне видны файлы которые TASK_00 удалил (`main.py`, `dashboard.py`, `cis-cli.py`, `src/`, `ENGINE/` и др.)
- Не видны новые файлы из TASK_01 (`LESSONS_LEARNED.md`, `PATTERNS.md`, `ANTIPATTERNS.md`)

Архитектор ошибочно сделал вывод что push не прошёл и поднял тревогу. После диагностики через `git log origin/main` и `git ls-tree -r --name-only main` на стороне Claude Code оказалось: **local main и remote main идентичны**, все коммиты на месте, legacy файлы реально удалены. Страница GitHub UI показывала закэшированное состояние недельной давности.

**Причина (почему так произошло):**
GitHub UI (или промежуточный CDN) кэширует рендер главной страницы репо. Свежие коммиты могут отображаться с задержкой до нескольких часов, особенно для репо с низкой активностью. `web_fetch` получает ровно то, что возвращает веб-сервер — закэшированную версию.

**Правильный подход:**
Для проверки реального состояния remote — использовать **git-операции**, а не рендер страницы:

```bash
# Архитектор может просить Claude Code выполнить:
git fetch origin
git log --oneline -10 origin/main
git log --oneline main..origin/main   # что в remote но не в local
git log --oneline origin/main..main   # что в local но не в remote
git ls-tree -r --name-only origin/main  # реальный список файлов в remote
```

Альтернатива через API (если `web_fetch` работает с api.github.com) — но в текущей среде архитектора `api.github.com` заблокирован.

**Правило на будущее:**
- **Не делать выводов** о состоянии remote только по рендеру главной страницы репо в web_fetch
- Для реальной проверки — запрашивать у Claude Code вывод git-команд (`git log`, `git ls-tree`, `git diff`)
- Если страница показывает устаревшее состояние — подождать 15-30 минут и обновить (cache TTL), либо проверить через git-команды
- При подозрении на расхождение **сначала** запросить диагностику у Claude Code, **потом** поднимать тревогу

**Применимость:** любой архитектурный ревью, любая сверка remote-состояния.
