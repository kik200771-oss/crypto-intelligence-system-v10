# TASKS/NOTES/ — Observability Channel

Эта папка — **легальный канал** для Claude Code передавать архитектору наблюдения и concerns **не нарушая контракт**.

Подробные правила — `CLAUDE.md § 31`. Ниже — быстрая справка.

---

## Когда писать сюда

В процессе выполнения TASK Claude Code замечает что-то **вне scope текущей задачи**:
- Паттерн в коде который повторяется
- Потенциальный риск который никто не адресовал
- Идея альтернативного подхода
- Недочёт в документации или тестах других модулей
- Что-то что стоит обсудить, но не прямо сейчас

Вместо того чтобы:
- **Молча исправить** (нарушение § 1-2, AP-07)
- **Блокировать работу через § 3** (слишком дорого для неблокирующих вопросов)
- **Забыть к концу задачи** (потеря ценной информации)

→ Записать сюда через один из шаблонов, продолжить работу, упомянуть в Reflection Block отчёта.

```

└── concerns/                       # Риски/проблемы с Impact
    └── YYYY-MM-DD_<slug>.md
```

## Observations vs Concerns — что куда

**Observation** — **описание** того что замечено:
- "В модулях ENGINE часто повторяется паттерн X"
- "Тесты в Y используют другой стиль чем в Z"
- "Есть несколько способов делать A в кодовой базе"

Тон: **нейтральный, описательный**. Без призыва к действию. Архитектор сам решит важно это или нет.

**Concern** — **обнаруженная проблема** с указанием Impact:
- "Compound status json содержит 5 компонентов needs_rewrite старше 2 недель"
- "В TASK_04 threshold z_score_threshold=5.0 математически недостижим на малых выборках"
- "В .gitignore правило logs/*.log не покрывает поддиректории"

Тон: **фактический, с оценкой severity**. Обязательное поле Impact (low/medium/high).

## Имена файлов

Формат: `YYYY-MM-DD_<brief_slug>.md`

Примеры хороших slug'ов:
- `2026-04-19_path_pattern_duplication.md` ✅
- `2026-04-19_stale_needs_rewrite_components.md` ✅
- `2026-04-19_zscore_threshold_math.md` ✅

Плохие slug'ы:
- `observation1.md` ❌ (не информативно)
- `important.md` ❌ (tag, не slug)
- `note_about_things.md` ❌ (вагон)

Slug максимум 4-5 слов, кратко описывает суть.

## Правила записи

1. **Использовать шаблон** — `TEMPLATE_observation.md` или `TEMPLATE_concern.md`. Не изобретать формат.
2. **Краткость** — 10-30 строк максимум. Если больше — разделить на несколько записей или решить что это § 3.
3. **Нейтральный тон** — факты, не эмоции; описание, не продажа идеи.
4. **Привязка к контексту** — указать какой TASK выполнялся когда замечено.
5. **Не дублировать с уроками** — если уже есть L-?? про это, сослаться, не повторять.
6. **Concerns обязательно с Impact** — low/medium/high. Если critical — это § 3, не NOTES.

## Что Observability НЕ делает

- **Не заменяет § 3** — блокирующие проблемы всё равно через § 3
- **Не разрешает AP-07** — самовольное добавление кода всё равно запрещено
- **Не обязует архитектора реагировать** на каждую запись
- **Не создаёт "мягкие разрешения"** обойти TASK

Запись сюда = "я хочу чтобы ты (архитектор) знал о X". Не = "тебе нужно решить X".

## Интеграция с Reflection Block (§ 9)

В финальном отчёте каждой задачи Claude Code указывает:
- Сколько observations/concerns записано (`Ideas logged: N`)
- Краткий index в Reflection Block

Пример:
```
Ideas logged: 2

## Observations from this task
- Замечено дублирование паттерна Path(__file__).resolve() в 8 модулях. 
  Детали в 2026-04-19_path_pattern_duplication.md.
- ...

## Self-critique
- Мог бы сверить API schema_validator с реальным кодом до написания теста.

## Questions for architect (non-blocking)
- Стоит ли вынести Path resolution в utils модуль?
  (см. concerns/2026-04-19_path_pattern_duplication.md)
```

---

## Интеграция с Cognitive Updates

Архитектор периодически (раз в ~5-10 задач) просматривает NOTES:
1. Группирует observations по темам
2. Если тема повторяется 3+ раз — кандидат на новый урок L-?? или AP-??
3. Если concern не решён — добавляется в backlog или в ближайший cleanup
4. Формализует через TASK_NN.5 cognitive_update

Таким образом NOTES — **источник материала** для эволюции когнитивной системы.

## Пример workflow

```
[День 1] Claude Code выполняет TASK_07
  → Замечает что в ENGINE/ много модулей используют 
    Path(__file__).resolve().parent.parent.
  → Записывает в observations/2026-04-22_path_pattern.md
  → Продолжает работу по TASK
  → В отчёте упоминает в Reflection Block.

[День 3] Claude Code выполняет TASK_09
  → Снова замечает тот же паттерн.
  → Обновляет existing observation (добавляет факт).

[День 5] Архитектор ревью'ет NOTES перед TASK_10
  → Видит что наблюдение встретилось 3 раза.
  → Создаёт TASK_10_utils_extraction: вынос Path resolution 
    в MARKET_MIND/utils/paths.py.
  → Или: формализует как новый P-?? паттерн + utility skeleton.
```

Без observations такой паттерн возможно был бы замечен, но **когда** — неизвестно.
